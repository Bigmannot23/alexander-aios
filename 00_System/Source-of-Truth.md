## Hierarchy

The newest document governs the **brain**; the prior documents govern the
**product**. Conflicts resolve by *scope*, not recency alone.

1. Latest explicit instruction in chat
2. **Maxed ClaudeOps Operator Blueprint** (newest) — brain/repo organization,
   learning refinery, IP/rights gate, claim-audit, source-quality, surfaces,
   measurement/cadence/pruning
3. **Ultimate Claude Code OS Stack** — multi-system + EntryLens engine
   architecture, product control-plane depth
4. **AlphaLab Live Vision Desk Playbook** — all live TradingView / vision /
   browser boundaries
5. **EntryLens Second Brain Addendum** — EntryLens doctrine, lanes, status
   semantics, templates
6. **Automation Leverage Blueprint** — automation ladder + six-system map
7. **Advanced Power Stack** — power-user Claude Code features
8. **Claude Code OS for EntryLens** — day-to-day repo workflow
9. **Nate Herk transcript** (non-binding philosophy, first youtube-ingest
   test case)
10. Older research

## Conflict rule

- Brain organization / learning / IP / surfaces → **#2**
- EntryLens product-code isolation + engine architecture → **#3–#5**
- Live chart / vision / browser → **#4**
- Unresolved conflicts become a decision log (`08_Decision-Logs/_index.md`),
  not a chat decision.

## Practical use

When in doubt about which document governs a question, find the *scope* of
the question first, then apply the table above. Document the resolution as
a decision log if it sets a precedent.
